# Conversion Runbook — 소스 무관 Mendix 컨버전 체크리스트

**이 문서는 스킬이 아니라 사용자용 실행 문서다.** 프로젝트 진행자(사람)가 각 단계의 프롬프트 블록을 순서대로 **그대로 복사해** 에이전트에 입력하면, 어떤 소스 스택이든 동일한 게이트를 거쳐 mxcli 기반 Mendix 앱으로 컨버전이 완료된다. 변수 치환은 필요 없다 — 경로는 아래 "고정 규약"의 명사로만 쓰며, 에이전트가 P단계에서 실제 경로를 확인해 프로젝트 CLAUDE.md에 기록한 뒤 스스로 해석한다. 프롬프트에는 스킬 경로를 적지 않는다 — 작업에 맞는 스킬을 찾아 읽는 것은 CLAUDE.md 라우팅 표에 따라 에이전트가 스스로 한다.

---

## 고정 규약 — 프롬프트에 나오는 명사의 의미 (치환 불필요)

| 명사 | 의미 | 에이전트가 해석하는 방법 |
|---|---|---|
| **워크스페이스** | 프로젝트 전용 폴더 — 에이전트 세션을 시작하는 현재 폴더. 툴킷 저장소와 분리 | 세션의 현재 폴더 |
| **툴킷** | `mxcli-project-toolkit` 클론 | 프로젝트 CLAUDE.md에 기록된 경로 (P-1에서 확인, P-3에서 기록) |
| **소스 폴더** | `sources/` 밑의 원본 소스 — 읽기 전용, 수정 금지 | `sources/` 아래 유일한 폴더 (여러 개면 에이전트가 사용자에게 질문) |
| **분석 폴더** | `analysis/<소스 폴더명>/` — 이 프로젝트의 모든 산출물 | 소스 폴더명에서 유도 |
| **대상 .mpr** | 대상 Mendix 프로젝트 파일 | 5단계 시작 시 에이전트가 워크스페이스에서 확인하고, 없으면 mxcli로 생성해 CLAUDE.md에 기록 — 사용자가 미리 준비할 필요 없음 |

**사용법:** 워크스페이스 루트에서 에이전트 세션을 열고, 프롬프트 블록을 위에서부터 순서대로 붙여넣는다. 복사·치환할 것이 없다. 진행 체크를 남기고 싶으면 이 파일을 워크스페이스에 복사해 체크박스만 표시한다(그 경우에도 치환은 불필요하며, 진행 기록은 툴킷 저장소에 커밋하지 않는다). P-1~P-3은 같은 세션에서 이어서 실행한다 — 프로젝트 CLAUDE.md가 생기기 전까지는 세션이 바뀌면 경로 기억이 없다.

## 런북 규칙 (모든 단계에 적용)

1. **스택 무언급** — 이 문서에는 특정 소스 기술명이 없다. 스택 판정과 그에 따른 분기는 0단계 산출물이 담당한다. 이 문서에 스택명을 적고 싶어진다면 그것은 인테이크/트리아지 산출물(`intake.md`/`triage.md`)에 들어갈 내용이다.
2. **프롬프트에는 작업만, 스킬 경로는 없음** — 프롬프트는 작업·산출물·게이트만 서술하고 블록 전체를 수정 없이 복사해 입력한다. 어떤 스킬을 읽을지는 프롬프트에 적지 않는다: 스킬 선택·로딩은 에이전트가 CLAUDE.md 라우팅(P-1에서 읽어두는 툴킷 라우팅 표 + P-3에서 생성되는 프로젝트 Baseline routing)으로 스스로 한다. 스킬이 개선·교체되어도 런북은 수정할 필요가 없다.
3. **게이트는 파일과 조건으로 판정** — "완료된 것 같음"이 아니라 "존재해야 하는 파일 + 그 안에 있어야 하는 내용"으로 판정한다. ✋ 표시는 **사용자 승인 없이는 다음 단계 진행 금지**인 하드 게이트다.
4. **0~4단계에서는 mxcli를 실행하지 않는다** — MDL/모델 조작은 5단계에서 시작한다. 잘못된 모듈 경계는 다이어그램 단계에서 고치는 것이 스크립트 40개가 그것을 전제한 뒤 고치는 것보다 훨씬 싸다.
5. **즉흥 판단 = 런북 결함** — 진행 중 이 문서에 없는 판단이 필요했다면, 그 프로젝트에서 우회하고 끝내지 말고 툴킷의 이 템플릿을 고친다(마무리 단계 참조).
6. **분기는 명시된 곳에서만** — B1(파이프라인 신규 구축), B2(문서 자료 유무), B3(멀티앱 질문) 세 분기 외의 단계 생략은 허용되지 않는다. 특히 "앱이 작으니 생략"은 금지 — 추출 파이프라인은 앱 크기와 무관하게 항상 세운다.
7. **범위의 기본값은 소스 전체다** — 런북은 "무엇을 변환할지"를 묻지 않는다. 소스에 있는 것은 전부 대상이다. 대신 묻는 것은 두 가지뿐: ① 소스가 참조하지만 소스 안에 없는 것(누락 의존성)을 어떻게 처리할지(추가 확보/스텁/미구현 선언), ② 소스가 한 번에 가기 큰 경우의 진행 순서(슬라이스는 순서이지 제외가 아니다).
8. **벤더 코드는 폴더 단위로 버리지 않는다** — 프레임워크/벤더 원형은 변환 대상이 아니라 Mendix 플랫폼이 그 역할을 대체하는 계층이지만, 벤더 폴더 안에는 프로젝트 커스터마이징이 섞여 있는 경우가 많다. 분류는 파일 단위 근거(저작권 헤더, 배포 원형 대조, 프로젝트 수정 흔적)로 하고, 원형도 삭제가 아니라 화면 정의·구문 해석용 레퍼런스로 유지한다.

### 단계 지도

| 순서 | 단계 | 하드 게이트 ✋ |
|---|---|---|
| P | 착수 준비 (워크스페이스·도구 점검 → CLAUDE.md·에이전트 → 인테이크) | — |
| 0 | TRIAGE (인벤토리 → 파이프라인 결정 → 경계 처리) | ✋ 파이프라인·경계 처리 서명 |
| 1 | ANALYSIS (A: 코드 추출 ∥ B: 문서 KB) | — |
| 2 | REQUIREMENTS (BRD 스캐폴드 → 보강 → 검증) | — |
| 3 | ARCHITECTURE & DESIGN (모듈 경계 → 청사진 ∥ 디자인) | ✋ (멀티앱 해소,) 모듈 경계 승인 |
| 4 | BUILD PLAN | ✋ 계획 승인 |
| 5 | BUILD (mxcli 시작: .mpr 확인/생성 → 빌드 루프) | 모듈별 빌드 루프 게이트 |
| 6 | TEST | 골든 패스·엣지·DB 어설션 |
| — | 마무리 (학습 환류) | — |

## P단계 — 착수 준비

### P-1. 워크스페이스 점검

- [ ] 프롬프트 실행

  ```
  지금 이 세션의 현재 폴더가 프로젝트 워크스페이스다. 아래 6가지를 점검해서
  항목별 통과/실패 표로 보고해줘. 실패가 하나라도 있으면 다음 진행을 멈추고 알려줘.

  1. mxcli-project-toolkit 클론(이하 "툴킷")의 위치를 찾아줘 — 프로젝트 CLAUDE.md에
     기록돼 있으면 그 경로, 없으면 ~/Mendix/mxcli-project-toolkit, 그래도 없으면
     나에게 물어봐. 찾으면 git pull로 최신화하고 마지막 커밋 한 줄을 기록한 뒤,
     툴킷 CLAUDE.md의 스킬 라우팅 표를 읽어둬 — 앞으로 모든 단계에서 작업에 맞는
     스킬을 이 라우팅으로 스스로 찾아 읽는다.
  2. 워크스페이스가 툴킷과 다른 경로의 자체 git 저장소인지 확인해줘.
  3. sources/ 밑의 소스 폴더를 확인하고 총 파일 수와 최상위 구조를 기록해줘.
     소스 폴더가 없거나 여러 개면 나에게 물어봐. 원본은 앞으로 절대 수정하지 않는다.
  4. analysis/<소스 폴더명>/ 을 만들어줘 (이하 "분석 폴더").
     하위 knowledge-base/는 파이프라인이 만들므로 미리 만들지 마.
  5. 워크스페이스 .gitignore에 node_modules/ 와 *.mpr.lock 이 있는지 확인하고,
     고객 원본(sources/)을 커밋할지 여부를 라이선스·보안 관점에서 나에게 물어봐줘.
     결정을 .gitignore와 함께 반영해줘.
  6. (소스를 툴킷 폴더 밑에 둔 변칙 구성일 때만) 툴킷 .gitignore가
     sources/, analysis/, knowledge-base/, *.mpr 를 제외하는지 확인해줘 —
     프로젝트 산출물이 툴킷에 커밋될 수 있는 상태면 진행 금지라고 보고해줘.
  ```

- [ ] **게이트:** 6개 항목 전부 통과 보고 (3번의 파일 수·구조 기록은 P-4 인테이크와 1-A 추출 커버리지 대조의 기준값이 된다)

### P-2. 도구 점검

- [ ] 프롬프트 실행

  ```
  컨버전에 필요한 도구를 점검해서 "지금 필수"와 "예약"으로 나눠 표로 보고해줘.
  지금 필수가 하나라도 없으면 멈추고 알려줘. 예약 항목은 지금 없어도 되고,
  현재 상태만 기록해줘 — 해당 단계 시작 전에 다시 점검한다.

  [지금 필수]
  - git --version
  - node --version (추출 파이프라인 실행용, LTS 권장)
  - npm --version (npm install은 0단계에서 파이프라인 확정 후 해당 pipeline/에서 실행)

  [예약 — 5단계(BUILD) 시작 전까지]
  - Mendix Studio Pro 설치 + 버전 기록 (5단계에서 .mpr을 확인/생성할 때 이 버전과 맞춘다)
  - mxcli --version (동작하면 툴킷의 알려진 mxcli 버그 로그에서 이 버전 관련 이슈 확인)
  - 대상 .mpr과 로컬 실행 URL은 여기서 점검하지 않는다 — 5단계 시작 시
    에이전트가 확인/생성/기록한다

  [예약 — 6단계(TEST) 시작 전까지]
  - npx playwright --version
  - 시드 투입·DB 어설션에 쓸 DB 접근 경로
  ```

- [ ] **게이트:** "지금 필수" 전부 통과. 예약 항목은 5단계·6단계의 전제 확인에서 재점검된다.

### P-3. 프로젝트 CLAUDE.md + 서브에이전트

- [ ] 프롬프트 실행

  ```
  이 워크스페이스의 프로젝트 CLAUDE.md를 툴킷 표준에 따라 생성해줘.
  반드시 포함할 것:
  ① 툴킷의 Baseline routing 표 (모든 MDL 작성 세션에 적용되는 상시 규율)
  ② "작업별 스킬 선택은 툴킷 CLAUDE.md의 라우팅 표를 따른다"는 지시 —
     이후 세션의 에이전트가 프롬프트에 스킬 경로가 없어도 스스로 찾게 하는 장치다
  ③ 경로 사실: 툴킷 경로, 소스 폴더, 분석 폴더, 대상 .mpr(이미 있으면 경로,
     없으면 "5단계 시작 시 mxcli로 생성"으로 표기하고 생성 시 갱신)
  프로젝트 고유 사실 섹션은 intake.md가 아직 없으므로 자리표시자로 두고,
  P-4 완료 후 갱신해야 한다고 명시해줘.
  ```

- [ ] 프롬프트 실행

  ```
  이 프로젝트의 개발 프로세스 서브에이전트(mdl 초안 / gate 검증 / test)를
  툴킷 표준에 따라 .claude/agents/ 밑에 생성해줘. 방금 만든 CLAUDE.md를
  먼저 읽고 시작해줘.
  ```

- [ ] 산출물 확인: `CLAUDE.md`(Baseline routing + 라우팅 지시 + 경로 사실 포함), `.claude/agents/*.md` 3종

### P-4. 인테이크 — 표준 8문항

- [ ] 프롬프트 실행

  ```
  소스 폴더를 탐색해서 아래 8개 질문에 답하는 intake.md를 분석 폴더에 작성해줘.
  코드로 확인되지 않는 항목은 추측으로 채우지 말고 "미확인 — 확인 경로: ..."로 남겨줘.

  1. 소스가 완전한 앱인가 발췌본인가? 그대로 컴파일/실행 가능한가?
  2. 소스가 참조하지만 소스 안에 없는 것(클래스·공통 모듈·화면·테이블 등 누락 의존성)의
     목록은? 각 항목의 처리 후보는? (추가 확보 / 스텁 / 미구현 선언 — 확정은 0단계 게이트에서.
     범위 자체는 묻지 않는다: 소스 전체가 대상이다)
  3. 데이터 모델의 출처는? (DDL / ORM 정의 / SQL·샘플 데이터에서 유추 / 별도 확보 필요)
  4. 시드·테스트용 샘플 데이터가 있는가? 어떤 파일인가?
  5. 프레임워크/벤더 레이어와 커스터마이징 레이어의 파일 단위 구분은? — 벤더 원형
     (추출 대상 아님, 화면 정의·위젯 시맨틱 해석용 레퍼런스로 유지)과 벤더 폴더 안에
     섞인 프로젝트 커스터마이징(추출 대상)을 나누고, 판정 근거(저작권 헤더, 배포 원형
     여부, 프로젝트 수정 흔적·주석)를 기록. 폴더 단위 일괄 제외 금지.
  6. 인증·권한 모델 정보가 소스 안에 있는가? (세션/역할/접근제어)
  7. 비즈니스 문서(설계서·사양서·매뉴얼·스프레드시트)가 있는가? 어디에?
  8. 미해결 질문을 답해줄 소스 시스템 담당자(SME)가 있는가?

  작성 후 CLAUDE.md의 프로젝트 고유 사실 섹션을 intake 결과로 갱신해줘.
  ```

- [ ] **게이트:** 분석 폴더의 `intake.md`에 8문항 전부가 "답" 또는 "미확인 — 확인 경로"로 기재됨. 빈 항목 금지. CLAUDE.md 갱신 완료.

## 0단계 — TRIAGE ✋

### 0-A. 수동 인벤토리

- [ ] 프롬프트 실행

  ```
  소스 폴더 전체를 마이그레이션 사전 평가 방법론에 따라 분석해서 분석 폴더에
  assessment.md를 작성해줘. 인벤토리 요약표와 6개 조사 영역 — 기술 스택,

  각 항목의 Mendix 매핑 열과 함께 정리해줘. intake.md 5번의 레이어 분류를 따라
  ```
